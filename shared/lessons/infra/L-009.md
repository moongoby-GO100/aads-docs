## 핵심 교훈

단일 watchdog는 자기 자신의 장애를 감지할 수 없다.
L1(자체방어) → L2(핵심감시) → L3(메타감시) → L4(외부감시) 4계층으로 구성하고,
복구 간 의존성을 그래프로 관리하며, 3단계 에스컬레이션을 적용해야 한다.
서버 간 상호 감시로 단일 장애점을 제거한다.

## 문제

- 단일 계층 감시(watchdog만 존재)는 watchdog 자신이 죽을 경우 아무것도 복구되지 않는다.
- 복구 작업 간 의존성(B가 먼저 복구되어야 A 복구 가능)을 무시하면 연쇄 실패 발생.
- 동일 서버에 반복 실패 시 쿨다운 없이 계속 재시도 → 리소스 고갈.
- MTTR 90분+: 단일 감시자 + 수동 개입 의존.

## 해결책

### L1: 프로세스 자체 방어
- 모든 프로세스에 hard_timeout(30분) 내장
- bridge 셀프체크: 60초 주기로 자기 상태 점검
- 타임아웃 발생 시 exit 124 → L2 트리거

### L2: 핵심 감시자
- watchdog_daemon: 30초 주기 서비스 모니터링
- pipeline_monitor: 2분 주기 파이프라인 흐름 감시
- recovery_graph: 의존성 위상정렬로 복구 순서 결정
- escalation_engine: 3단계 에스컬레이션 (L2→L3→L4)
- circuit_breaker: 3회 연속 실패 시 5분 쿨다운

### L3: 메타 감시자
- meta_watchdog.sh: cron */1 (서버 211)
- L2 감시자들의 생존 여부 확인
- L2 장애 시 재시작 + 텔레그램 긴급 알림

### L4: 외부 감시
- UptimeRobot / GitHub Actions
- 서버 자체가 응답 불능일 때 외부에서 감지

### 서버 상호 감시
- 3서버(211, 68, 114) 삼각형 크로스 모니터링
- 2분 주기, cross_monitor.sh
- 1대 장애 시 나머지 2대가 자동 복구

## 결과

- Before: 단일계층 감시, 12건 복구(연쇄실패), MTTR 90분+
- After: 4계층 감시, 15건 복구(의존성관리), 서버상호감시, MTTR 5분 이내
- 비용: $0 (스크립트+무료서비스)
