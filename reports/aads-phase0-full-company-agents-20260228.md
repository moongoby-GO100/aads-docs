## 1. 비전: AADS는 스스로 성장하는 AI 회사다

v2.0에서는 8개 본부 43개 에이전트를 정의했다.
그러나 이것은 **초기 인재 풀**이지 고정 조직이 아니다.

실제 회사가 새로운 사업 영역에 진입할 때 조직을 확장하듯이,
AADS도 프로젝트 요구에 따라 **본부를 신설하고, 에이전트를 자동 생성하고, 조직을 확장**한다.

핵심 원칙:

```
1. 본부/에이전트 수는 고정이 아니다 — 총괄 AI가 필요에 따라 확장한다.
2. 43개는 시드(seed)이다 — 프로젝트를 수행할수록 조직이 커진다.
3. 새 에이전트는 AI가 AI를 만든다 — 총괄 AI가 역할, 프롬프트, 도구를 자동 정의한다.
4. 성과 없는 에이전트는 퇴출된다 — HR 매니저가 비활성 에이전트를 아카이브한다.
```

## 2. 동적 조직 확장 시스템 (Agent Factory)

### 2.1 개념

총괄 AI가 프로젝트를 분석할 때, 기존 레지스트리에서 필요한 에이전트를 검색한다.
**적합한 에이전트가 없으면**, 총괄 AI가 Agent Factory를 호출하여 새 에이전트를 생성한다.

```
사용자: "패션 브랜드 D2C 쇼핑몰, 브랜딩이 핵심"
    │
    ▼
[총괄 AI] 프로젝트 분석
    │
    ├── 필요 역량: 브랜드 디자인, 비주얼 디렉션, 룩북 기획
    ├── 레지스트리 검색 → design-uiux-001 (UI/UX 디자이너) 1명만 존재
    ├── 판단: "브랜드 디자인 전문가 없음. 생성 필요."
    │
    ▼
[Agent Factory] 새 에이전트 자동 생성
    │
    ├── 1. 역할 정의: 브랜드 디자이너
    ├── 2. 시스템 프롬프트 자동 작성
    ├── 3. 전문 분야·태그·모델 배정
    ├── 4. 레지스트리 등록: design-brand-001
    ├── 5. 테스트 호출 (온보딩)
    │
    ▼
[총괄 AI] 새 에이전트를 팀에 배정
```

### 2.2 Agent Factory 동작 흐름

```python
# 의사 코드 — core/agents/agent_factory.py

class AgentFactory:
    """총괄 AI가 필요한 에이전트를 자동 생성하는 공장"""

    async def create_agent(self, requirement: AgentRequirement) -> AgentCard:
        """
        requirement 예시:
        {
            "role": "designer",
            "specialty_needed": "브랜드 디자인, 비주얼 아이덴티티, 컬러 팔레트",
            "context": "패션 D2C 쇼핑몰 프로젝트, 20대 여성 타겟",
            "reference_agents": ["design-uiux-001"]  # 참고할 기존 에이전트
        }
        """

        # 1단계: 총괄 AI에게 에이전트 카드 생성을 요청
        agent_card = await self.supervisor.generate_agent_card(
            prompt=f"""
            다음 역량이 필요한 새로운 에이전트를 설계하세요.

            필요 역량: {requirement.specialty_needed}
            프로젝트 컨텍스트: {requirement.context}
            참고 에이전트: {requirement.reference_agents}

            다음 JSON 형식으로 에이전트 카드를 출력하세요:
            {{
                "agent_id": "자동생성 (역할-전문분야-번호)",
                "name": "한글 직함",
                "role": "역할 카테고리",
                "specialty": ["전문 분야 배열"],
                "description": "이 에이전트가 무엇을 하는지 상세 설명",
                "system_prompt": "이 에이전트의 시스템 프롬프트 (500자 이상)",
                "model_tier": {{"default": "모델명", "fallback": "모델명"}},
                "tools": ["필요한 도구"],
                "tags": ["검색용 태그"],
                "division": "소속 본부 (기존 본부 또는 신설 본부명)",
                "auto_generated": true,
                "generated_for_project": "프로젝트 ID",
                "generated_at": "타임스탬프"
            }}
            """
        )

        # 2단계: 에이전트 카드 유효성 검증
        validated = self.validate_agent_card(agent_card)

        # 3단계: 소속 본부 확인 — 기존 본부에 배정 또는 신설
        division = self.resolve_division(validated.division)

        # 4단계: 레지스트리에 등록
        self.registry.register_agent(validated)

        # 5단계: 온보딩 테스트 (간단한 작업을 수행시켜 응답 품질 확인)
        test_result = await self.onboarding_test(validated)
        if test_result.quality_score < 0.6:
            # 프롬프트 보강 후 재시도
            validated = await self.refine_prompt(validated, test_result)

        # 6단계: HR 매니저에게 등록 통보
        await self.hr_manager.notify_new_agent(validated)

        return validated
```

### 2.3 본부(Division) 동적 신설

에이전트가 생성될 때 기존 8개 본부에 맞지 않으면, **새 본부를 자동 신설**한다.

```python
def resolve_division(self, division_name: str) -> Division:
    """기존 본부에 매칭되면 배정, 없으면 신설"""

    existing = self.registry.get_division(division_name)
    if existing:
        return existing

    # 새 본부 생성
    new_division = Division(
        id=f"div-{slugify(division_name)}",
        name=division_name,
        created_by="agent-factory",
        created_at=now(),
        agents=[]
    )
    self.registry.register_division(new_division)

    # COO에게 통보
    self.coo.notify_new_division(new_division)

    return new_division
```

**실제 시나리오:**

| 프로젝트 요청 | 필요 에이전트 | 본부 변화 |
|---------------|-------------|-----------|
| "패션 D2C 쇼핑몰, 브랜딩 핵심" | 브랜드 디자이너, 비주얼 디렉터, 포토그래퍼 가이드 | **디자인본부 신설** (기존 설계본부의 UI/UX와 별도) |
| "K-뷰티 일본 진출 마케팅" | 일본 시장 전문가, SNS 크리에이터, 인플루언서 마케팅, 카피라이터 | **마케팅본부 확장** (기존 사업본부 내 마케팅팀 → 독립 본부) |
| "의료 AI 진단 서비스" | 의료 AI 전문가, 임상 데이터 전문가, 의료기기 규제 전문가 | **의료 전문 본부 신설** |
| "메타버스 교육 플랫폼" | 3D 개발자, 교육 설계 전문가, 게임 디자이너 | **에듀테크본부 신설** |
| "블록체인 DeFi 플랫폼" | 스마트 컨트랙트 전문가, 토큰 이코노미 설계, Web3 프론트엔드 | **Web3본부 신설** |

### 2.4 에이전트 생명주기

에이전트는 생성되고, 활동하고, 성장하고, 퇴출된다.

```
[탄생] Agent Factory가 생성
    │
    ▼
[온보딩] 테스트 작업 수행 → 품질 확인
    │
    ▼
[활동] 프로젝트에 투입, 작업 수행
    │
    ├── 성과 좋음 → 우선 순위 점수(priority_score) 상승
    ├── 성과 나쁨 → 프롬프트 개선 시도
    └── 프롬프트 개선 후에도 성과 나쁨 → 모델 업그레이드 시도
    │
    ▼
[성장] 누적 프로젝트 경험 → 프롬프트에 "이전 프로젝트 경험" 추가
    │
    ▼
[휴면] 90일간 투입 없음 → 비활성 상태
    │
    ▼
[퇴출/아카이브] HR 매니저 판단 → 레지스트리에서 제거 (기록은 보존)
```

```yaml
# 에이전트 카드에 추가되는 생명주기 필드
lifecycle:
  status: active          # active | dormant | archived
  created_at: 2026-03-05
  last_used: 2026-03-20
  total_projects: 7
  success_rate: 0.86       # 성공적 산출물 비율
  avg_cost_per_task: 0.23  # 평균 작업당 비용
  priority_score: 0.88     # 높을수록 우선 배정
  prompt_version: 3        # 프롬프트 개선 횟수
  model_upgrades: 1        # 모델 업그레이드 횟수
```

```
총괄 AI가 프로젝트를 분석한 후:

1. 필요 역량 목록 추출
   예: ["브랜드 디자인", "PG결제", "React Native", "일본어 마케팅"]

2. 각 역량에 대해 레지스트리 검색

   ├── "React Native" → dev-mobile-001 (유사도 0.91) → ✅ 있음
   └── "일본어 마케팅" → 검색 결과 0건 → ❌ 없음

3. 없는 역량에 대해 Agent Factory 호출
   ├── "브랜드 디자인" → Agent Factory → design-brand-001 생성
   └── "일본어 마케팅" → Agent Factory → sales-marketing-jp-001 생성

4. 팀 편성 (기존 + 신규 혼합)
   [PM, dev-payment-001, dev-mobile-001, design-brand-001(NEW), sales-marketing-jp-001(NEW), QA]
```

## 5. 에이전트 간 상호작용 흐름

### 5.1 프로젝트 수행 흐름 (v3.0 — Agent Factory 포함)

```
대표님: "○○○ 만들어"
    │
    ▼
[총괄 AI] 프로젝트 분석
    │
    ├─→ [시장조사 애널리스트] 경쟁사·시장 데이터
    ├─→ [비즈니스 애널리스트] 사업 타당성
    ├─→ [리스크 매니저] 초기 리스크 평가
    │
    ▼
[총괄 AI] 필요 역량 추출 → 레지스트리 검색
    │
    ├── 있는 역량 → 기존 에이전트 배정
    ├── 없는 역량 → [Agent Factory] 새 에이전트 생성
    │                   ├── 에이전트 카드 자동 생성
    │                   ├── 레지스트리 등록
    │                   ├── [HR 매니저] 온보딩 테스트
    │                   └── 본부 배정 또는 신설
    │
    ├─→ [HR 매니저] 에이전트 가용성·역량 최종 확인
    ├─→ [일정 관리자] 실행 일정 최적화
    ├─→ [비용 감시자] 예상 비용 → [CFO] 예산 승인
    │
    ▼
[총괄 AI] 동적 그래프 생성 → 파이프라인 실행
    │
    ├── 각 Phase 실행 (기존 + 신규 에이전트 혼합)
    │
    ▼ (각 Phase 완료 시 체크포인트)
    │
    ├─→ [비용 감시자] 비용 점검
    ├─→ [문서 관리자] 산출물 아카이빙
    ├─→ [리스크 매니저] 리스크 재평가
    ├─→ [총괄 AI] 필요 시 팀 재구성 (에이전트 교체/추가/제거/신규 생성)
    │
    ▼
프로젝트 완료
    │
    ├─→ [재무 매니저] 최종 비용 정산
    ├─→ [HR 매니저] 에이전트 성과 평가 (신규 에이전트 포함)
    ├─→ [문서 관리자] 최종 보고서 + 지식 베이스 갱신
    └─→ 신규 에이전트: 레지스트리에 영구 보존 (다음 프로젝트에서 재활용 가능)
```

새 프로젝트에서 더 정확한 팀을 편성할 수 있다.
