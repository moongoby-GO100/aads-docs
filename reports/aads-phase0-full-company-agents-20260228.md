## 1. 비전: AADS는 "AI 회사" 그 자체다

이전 설계에서는 개발 에이전트 20개(PM, Architect, Developer 등)만 정의했다.
그러나 실제 회사는 개발팀만으로 돌아가지 않는다.

영업이 고객을 데려오고, CS가 고객을 관리하고, 재무가 비용을 통제하고,
법무가 리스크를 차단하고, 마케팅이 시장을 만든다.

AADS의 최종 형태는 **"AI가 운영하는 완전한 소프트웨어 회사"** 이다.
대표님(CEO)이 지시를 내리면, AI C-Level 임원들이 각 부서를 운영하고,
각 부서의 전문 에이전트들이 실무를 수행한다.

```
                        ┌─────────────┐
                        │  대표님(CEO) │
                        │  (인간, 최종 의사결정) │
                        └──────┬──────┘
                               │
                    ┌──────────┼──────────┐
                    │          │          │
              ┌─────┴──┐ ┌────┴───┐ ┌────┴───┐
              │총괄 AI  │ │  CTO   │ │  COO   │
              │Supervisor│ │AI 기술 │ │AI 운영 │
              └────┬───┘ └───┬────┘ └───┬────┘
                   │         │          │
    ┌──────────────┼─────────┼──────────┼──────────────┐
    │              │         │          │              │
┌───┴───┐   ┌─────┴──┐ ┌───┴───┐ ┌───┴───┐   ┌─────┴──┐
│개발본부│   │기획본부│ │경영지원│ │사업본부│   │품질관리│
│Dev Div │   │Plan Div│ │Biz Ops│ │Biz Dev│   │QA Div  │
└───────┘   └────────┘ └───────┘ └───────┘   └────────┘
```

**총괄 AI와 C-Level의 관계:**
- 총괄 AI가 프로젝트 단위 의사결정 (팀 편성, 실행 관리)
- CTO/COO/CFO는 전사 단위 의사결정 (기술 방향, 운영 효율, 재무 건전성)
- 대표님은 최종 의사결정 + 전략 방향 설정

## 5. 가동 모드 분류

모든 에이전트가 항상 돌면 비용이 폭증한다.
가동 모드를 3단계로 나누어 비용을 최적화한다.

| 가동 모드 | 설명 | 해당 에이전트 | 비용 구조 |
|-----------|------|---------------|-----------|
| **상시 가동 (Always-On)** | 24/7 모니터링, 실시간 반응 필요 | 비용 감시자, 운영 매니저, 문서 관리자, 일정 관리자 | 경량 모델(Gemini Flash) + 이벤트 기반 호출로 비용 최소화 |
| **프로젝트 가동 (Project-Active)** | 프로젝트 실행 시에만 활성화 | 개발본부 전원, 기획본부, 설계본부, QA, 인프라, 도메인 전문가 | 프로젝트 비용에 포함 |
| **요청 가동 (On-Demand)** | 대표님 또는 다른 에이전트가 호출할 때만 | 영업, CS, 마케팅, 그로스, 파트너십, 법무, HR, 리스크, 시장조사 | 호출 건당 과금 |

상시 가동 에이전트의 비용을 최소화하기 위해:
- 비용 감시자: Gemini Flash + 30초 간격 폴링 → 일일 비용 약 $0.05
- 운영 매니저: 5분 간격 상태 체크 → 일일 약 $0.02
- 문서 관리자: 이벤트 트리거(산출물 생성 시에만) → 프로젝트당 약 $0.10
- 일정 관리자: 1시간 간격 스캔 → 일일 약 $0.01

**상시 가동 일일 비용: 약 $0.08 (월 $2.4)**

## 6. 비용 감시 시스템 상세 설계

대표님이 특별히 언급하신 비용 감시자의 동작을 상세히 정의한다.

### 6.1 감시 규칙

```yaml
# core/agents/registry/biz-cost-001.yaml 내 rules 섹션

cost_rules:
  # 단일 호출 경보
  single_call_warning: 3.00    # $3 이상 → 경고
  single_call_critical: 5.00   # $5 이상 → 즉시 차단 검토

  # 일일 경보
  daily_warning: 30.00         # 일일 $30 → 경고
  daily_critical: 50.00        # 일일 $50 → 총괄AI에게 모델 다운그레이드 권고

  # 프로젝트 예산 경보
  project_budget_warning: 0.80  # 예산 80% 도달 → 경고
  project_budget_critical: 0.95 # 예산 95% 도달 → 작업 일시 중지, 대표님 승인 필요

  # 전사 월간 예산 경보
  monthly_budget_warning: 0.70  # 70% → 경고
  monthly_budget_critical: 0.90 # 90% → 신규 프로젝트 시작 보류

  # 이상 탐지
  spike_detection: true         # 전일 대비 300% 이상 급증 시 경보
  idle_cost_detection: true     # 에이전트가 유의미한 산출물 없이 비용만 발생 시 경보

actions:
  warning: "로그 기록 + 총괄 AI 알림"
  critical: "로그 기록 + 총괄 AI 알림 + 대표님 알림 + 자동 모델 다운그레이드 권고"
  budget_exceeded: "작업 일시 중지 + 대표님 승인 대기"
```

### 6.2 비용 감시자 → 재무 매니저 → CFO 보고 체인

```
[비용 감시자] 실시간 감시
    │
    ├── 정상: 30초마다 비용 집계, 아무 조치 없음
    ├── 경고: [총괄 AI]에게 알림 → 모델 다운그레이드 검토
    └── 위험: [대표님]에게 직접 알림 + 작업 중지
    │
    ▼ (매일 23:59)
[재무 매니저] 일일 정산
    │
    ├── 프로젝트별 비용 분해
    ├── 에이전트별 비용 분해
    ├── 모델별 비용 분해
    └── 리포트 생성 → Redis 저장 + 대시보드 표시
    │
    ▼ (매주 금요일)
[CFO] 주간 재무 리뷰
    │
    ├── 수익성 분석 (프로젝트 매출 vs 비용)
    ├── 예산 대비 실적
    └── 다음 주 예산 배정 → 대표님 리포트
```

### 7.2 활용 방식

에이전트가 작업 중 필요한 컨텍스트를 요청하면, 문서 관리자가 벡터 검색을 수행하여 제공한다.

```
[Developer 에이전트]: "이전에 토스페이먼츠 API 연동한 코드 있어?"
    │
    ▼
[문서 관리자]: Qdrant 벡터 검색 → "project-xyz에서 dev-payment-001이 생성한
                payment_service.py 발견. 유사도 0.94"
    │
    ▼
[Developer 에이전트]: 해당 코드를 참조하여 새 프로젝트에 적용
```

이것이 AADS의 **학습 효과**가 된다.
프로젝트를 할수록 지식 베이스가 축적되어 다음 프로젝트가 더 빨라지고 품질이 올라간다.
